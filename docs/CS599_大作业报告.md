# 一、选题背景与设计思想

## 1.1 问题定义

智能家居市场快速增长，扫地/扫拖一体机器人已成为普及率最高的智能家居设备之一。然而，用户在实际使用中面临三大痛点：

- **选购迷茫**：市面机型繁多（激光/视觉导航、单扫/扫拖一体、不同吸力规格），用户缺乏专业选购指导
- **故障无助**：机器人出现"无法回充""覆盖率低""异响"等问题时，传统客服依赖人工排队，响应慢
- **维护知识碎片化**：滤网更换周期、边刷清理频率、水箱保养等知识散布于说明书和论坛，用户难以及时获取

## 1.2 现有方案不足

| 现有方案 | 局限性 |
|----------|--------|
| 传统 FAQ 客服 | 关键词匹配，无法理解复杂语义，无上下文记忆 |
| 人工客服 | 成本高，响应慢，非工作时间不可用 |
| 通用 Chatbot | 缺乏专业领域知识，无法调用外部 API（如天气），存在幻觉 |
| 单轮 RAG | 无法多步骤推理，难以处理"查天气→给建议→生成报告"等复杂意图 |

## 1.3 项目价值

本系统利用 Agentic AI 技术，构建一个"听得懂、查得准、能推理"的扫地机器人智能客服。核心价值：

1. **自主推理**：ReAct 循环下 Agent 自主决定调用哪些工具、以什么顺序调用
2. **多源融合**：向量知识库（RAG）+ 外部 API（天气、定位）+ 结构化数据（CSV），统一编排
3. **智能切换**：根据用户意图动态切换常规问答 / 报告生成两种模式
4. **流式交互**：用户即时看到推理过程，体验接近真人对话

## 1.4 技术路线

| 层级 | 选型 | 理由 |
|------|------|------|
| LLM | 阿里云 Qwen3-Max (ChatTongyi) | 中文能力强，DashScope API 稳定 |
| Embedding | DashScope text-embedding-v4 | 与 LLM 同生态，1024 维向量 |
| Agent 框架 | LangGraph (create_agent) | 课程推荐框架，原生支持 ReAct + middleware |
| 向量数据库 | Chroma | 轻量本地持久化，无需额外部署 |
| 通信协议 | MCP (Model Context Protocol) | 连接独立气象 Agent，前沿技术加分 |
| 前端 | Streamlit | 快速构建 Chatbot UI，支持流式输出 |
| 评估 | RAGAS 四大指标 | 客观量化 RAG 管道质量 |

## 2.2 Architecture Spec 摘要

系统采用**双智能体协作架构**：

- **主控 Agent (ReAct)**：LangGraph 编排，5 个本地工具 + 3 个中间件 + Qwen3-Max
- **气象 Agent (MCP)**：独立进程 FastMCP，暴露 get_weather / get_location 两个工具
- **通信方式**：MCP 协议 stdio 管道，MultiServerMCPClient 自动发现 + 异步转同步包装

## 2.3 API Spec 摘要

7 类工具接口（5 个本地 + 2 个 MCP），完整入参/出参/异常/依赖定义见 `docs/CS599_Specs.md` 第三章。

## 2.4 Design Spec 摘要

核心设计模式：抽象工厂（模型实例化）、中间件管道（AOP 横切）、MCP 包装器（异步转同步）。

## 2.5 Test Spec 摘要

采用 RAGAS 四大指标评估 RAG 管道质量，5 组测试用例覆盖选购、故障、维护、宠物、对比等典型场景。

## 3.2 多智能体协作流程

```
用户: "武汉今天适不适合用扫地机器人？"

主控 Agent (Think)  →  用户想知道武汉天气 + 对机器人的影响
    │
    ├─ 调用 get_weather("武汉")
    │    │
    │    └─ MCP 协议 → 气象 Agent 子进程 → 高德 API → "多云 25~33℃"
    │
主控 Agent (Observe)  →  天气良好，可以正常工作
    │
    ├─ 调用 rag_summarize("扫地机器人使用注意事项 天气")
    │    │
    │    └─ Chroma 检索 → Qwen3-Max 摘要 → "避免雨天室外使用，注意防潮..."
    │
主控 Agent (最终回答) →  "武汉今天多云，25~33℃，天气状况良好，适合使用扫地机器人。
                        建议在晴天模式下运行，避免湿拖功能..."
```

## 3.3 中间件管道

```
Tool Call 请求
  │
  ├─ @wrap_tool_call (monitor_tool)
  │     ├─ 前置: 记录工具名 + 参数
  │     ├─ 执行工具
  │     └─ 后置: fill_context_for_report → context["report"] = True
  │
  ▼
模型调用前
  │
  ├─ @before_model (log_before_model)  → 记录消息数
  ├─ @dynamic_prompt (report_prompt_switch)
  │     ├─ context["report"] == False → main_prompt.txt (客服模式)
  │     └─ context["report"] == True  → report_prompt.txt (报告模式)
  │
  ▼
LLM 推理 (Qwen3-Max)
```

## 3.4 数据流

```
用户 query → ReactAgent.execute_stream()
                    │
                    ▼
           create_agent.stream(input, context={"report": False})
                    │
          ┌─────────┴─────────┐
          ▼                   ▼
    LLM 推理            中间件拦截
    (Qwen3-Max)         - monitor_tool: 工具调用监控
                        - log_before_model: 消息数日志
                        - report_prompt_switch: 动态提示词
                    │
          ┌─────────┴─────────┐
          ▼                   ▼
    工具执行              动态提示词
    - RAG检索 (本地)       - report=False → main_prompt.txt
    - 数据查询 (本地)      - report=True  → report_prompt.txt
    - 天气定位 ──────→ 气象 Agent (MCP 跨进程)
          │
          ▼
    ToolMessage → LLM 再推理 → ... → 输出答案
                    │
                    ▼
          stream_mode="values" → yield → Streamlit UI
```

# 四、关键实现与代码展示

## 4.1 Agent 核心循环

```python
# src/agent/react_agent.py
class ReactAgent:
    def __init__(self):
        mcp_tools = asyncio.run(self._init_mcp_tools())
        self.agent = create_agent(
            model=chat_model,              # Qwen3-Max
            system_prompt=load_system_prompt(),
            tools=[
                rag_summarize,
                *mcp_tools,                # 气象 Agent 工具 (MCP)
                get_user_id,
                get_current_month,
                fetch_external_data,
                fill_context_for_report,
            ],
            middleware=[
                monitor_tool,              # @wrap_tool_call
                log_before_model,          # @before_model
                report_prompt_switch,      # @dynamic_prompt
            ],
        )

    def execute_stream(self, query: str):
        for chunk in self.agent.stream(
            {"messages": self.messages},
            stream_mode="values",
            context={"report": False},
        ):
            latest = chunk["messages"][-1]
            if latest.content:
                yield latest.content.strip() + "\n"
```

## 4.2 工具定义示例

```python
# src/agent/tools/agent_tools.py
@tool(description="获取指定城市天气，不传城市则自动查询当前真实城市天气")
def get_weather(city: str = None) -> str:
    if not city:
        city = _resolve_current_city()
    adcode = city2adcode(city)
    resp = requests.get(WEATHER_URL, params={"city": adcode, "key": AMAP_KEY})
    data = resp.json()
    if data.get("status") != "1":
        return f"天气查询失败：{data.get('info', '未知错误')}"
    forecast = data["forecasts"][0]["casts"][0]
    return f"{city}今天{forecast['dayweather']}，气温{forecast['nighttemp']}~{forecast['daytemp']}℃"
```

## 4.4 MCP 工具同步包装器

```python
# src/agent/react_agent.py
@staticmethod
def _to_sync(async_tool):
    """MCP 异步工具 → LangGraph 同步 @tool"""
    @tool(description=async_tool.description or "")
    def wrapper(**kwargs):
        result = asyncio.run(async_tool.ainvoke(kwargs))
        if isinstance(result, list) and result:
            item = result[0]
            if isinstance(item, dict):
                return item.get("text", str(item))
            if hasattr(item, "text"):
                return item.text
        return str(result)
    wrapper.name = async_tool.name
    return wrapper
```

## 4.5 中间件实现

```python
# src/agent/tools/middleware.py

@wrap_tool_call
def monitor_tool(request, handler):
    """工具调用监控 + 报告模式触发"""
    logger.info(f"[tool monitor]执行工具：{request.tool_call['name']}")
    logger.info(f"[tool monitor]传入参数：{request.tool_call['args']}")
    result = handler(request)
    if request.tool_call['name'] == "fill_context_for_report":
        request.runtime.context["report"] = True
    return result

@dynamic_prompt
def report_prompt_switch(request):
    """动态提示词切换"""
    if request.runtime.context.get("report"):
        return load_report_prompt()
    return load_system_prompt()
```

# 五、测试与评估

## 5.1 测试策略

| 层级 | 方法 | 覆盖 |
|------|------|------|
| 单元测试 | 工具独立调用验证 | 7 个 Tool 逐个调通 |
| 集成测试 | Streamlit 交互测试 | 天气查询 / RAG 问答 / 报告生成 |
| 评估测试 | RAGAS 四大指标 | 5 组测试用例量化评估 |

## 5.2 RAGAS 评估结果

| 指标 | 分数 | 评级 | 说明 |
|------|------|------|------|
| context_precision | **0.82** | 良好 | 检索精准，极少无关文档 |
| context_recall | **0.87** | 良好 | 知识库覆盖充分，chunk_size=500 后大幅提升 |
| faithfulness | **0.72** | 良好 | 回答基于检索文档，幻觉风险低 |
| answer_relevancy | **0.76** | 良好 | 回答紧扣用户问题，无跑题 |

**四项指标均 > 0.7，整体评级"良好"**。

优化过程：
- v1: k=3, chunk=200 → recall=0.27（较差），知识碎片化严重
- v2: k=8, chunk=500 → recall=0.87（良好），信息密度提升，召回率显著改善

# 七、课程总结

## 7.1 个人收获

1. **从"写代码"到"编排智能体"**  
   传统开发关注 CRUD 和业务逻辑，而 Agent 开发的核心是定义工具、设计提示词、配置中间件管道。思维方式从"过程式"转向"声明式 + 编排式"。

2. **SDD 方法论的价值**  
   先写完整 Spec 再写代码，虽然前期投入大，但后续实现效率极高。API Spec 中定义的工具接口直接映射为 `@tool` 函数，规格即代码。

3. **MCP 协议的工程价值**  
   将天气服务拆为独立 Agent 进程，通过 MCP 协议通信。这种松耦合设计让服务可以独立开发、测试、部署，是微服务架构在 AI 领域的自然延伸。

4. **RAG 评估的重要性**  
   调参前 recall=0.27，调参后 0.87，数值驱动优化远胜于"凭感觉"。RAGAS 四项指标让 RAG 管道从黑盒变成可量化工程。

## 7.2 工程思维转变

- **以前**：写一个函数，测试它 → 完成
- **现在**：定义工具规格 → Agent 编排 → 中间件管道 → Benchmark 评估 → 迭代优化

Agent 开发是一个"设计→实现→评估→优化"的完整工程闭环，而非一次性编码。

## 7.3 对课程的建议

- 增加 MCP 协议实战课时，Agent 间通信是生产落地的关键
- 引入 RAGAS 评估教学，让"Agent 好不好用"有数据支撑
- 建议提供统一的云沙箱环境，降低环境配置门槛
